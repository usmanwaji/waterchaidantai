## สรุปผู้บริหาร (Executive Summary)

ระบบนี้ครอบคลุมวงจรจัดการอุทกภัยได้กว้างผิดปกติสำหรับโปรเจคขนาดนี้: เฝ้าระวัง (map) → คาดการณ์ (forecast) → วิเคราะห์ย้อนหลัง (history/repeat) → จำลอง (sim) → ตอบโต้ (shelter/route/resources) → บริหารสมาชิก (admin) ถือว่าโครง "ecosystem" ครบกว่าหลายระบบของหน่วยงานรัฐจริง

**ช่องว่างใหญ่สุด 3 ข้อ (ถ้าทำแค่ 3 อย่าง ให้ทำ 3 อย่างนี้):**

| # | ช่องว่าง | เหตุผล |
|---|---------|--------|
| 1 | **ไม่มีระบบแจ้งเตือนเชิงรุก (Push Alert)** | ทุกหน้าเป็น pull — ผู้ใช้ต้องเปิดเว็บเอง แต่น้ำท่วมฉับพลันเกิดตี 2 ระบบที่ไม่ปลุกคนตอนตี 2 ยังไม่ใช่ระบบเตือนภัย |

# ส่วนที่ 1: วิเคราะห์รายหน้า (Page-by-Page)

## 1.1 index.html — ภาพรวมอากาศ & สถานการณ์น้ำ

**ที่มีอยู่:** พยากรณ์รายชั่วโมง 24 ชม. + รายวัน 7 วัน (Open-Meteo), ฝนสะสม 3 วัน/7 วัน, อำเภอเฝ้าระวัง, มาตรวัดน้ำต้องระวัง (DDPM NWT02), สถานีน้ำท่าหลัก, เรดาร์สทิงพระ, แผนที่ WRF-ROMS — เลือกจังหวัด/อำเภอได้ ดีมากสำหรับหน้าแรก

**ควรเพิ่ม:**

- **🚦 Risk Banner รวมด้านบนสุด** — วิดเจ็ตเดียวตอบคำถามแรกของทุกคน: "อำเภอฉันเสี่ยงแค่ไหนตอนนี้" คำนวณจาก 3 ปัจจัยที่มีข้อมูลอยู่แล้ว: (ก) ฝนพยากรณ์ 24 ชม. (ข) สถานีน้ำใกล้เคียงที่เกิน 80% ตลิ่ง (ค) ดินอิ่มน้ำจาก rain3d
  ```
  ┌─────────────────────────────────────────────┐
  │ 🟠 อ.สุไหงโก-ลก: เฝ้าระวัง (ระดับ 2/4)        │
  │ ฝน 24 ชม. 85 มม. · สถานี X011 เหลือ 0.8 ม.   │
  │ ถึงตลิ่ง · ดินอิ่มน้ำ 78%    [ดูรายละเอียด →] │
  └─────────────────────────────────────────────┘
  ```
  4 ระดับตามเกณฑ์ ปภ.: 🟢 ปกติ / 🟡 เฝ้าระวัง / 🟠 เตรียมอพยพ / 🔴 อพยพ
- **🌧️ Nowcast 3 ชม. จากเรดาร์** — คุณมี API เองอยู่แล้ว (gain9999/nowcast แปลงเรดาร์กรมอุตุฯ เป็น JSON ระบุ lat/lon ได้) แต่หน้าแรกยังแสดงเรดาร์เป็น "ภาพนิ่ง" ให้ดึง nowcast มาแสดงเป็นแถบ: "ฝนจะถึง อ.ที่เลือก ในอีก ~40 นาที ความแรงปานกลาง" — นี่คือ quick win ที่คุ้มที่สุดเพราะ backend เสร็จแล้ว
- **ฝนตกจริง (observed) เทียบพยากรณ์** — ตอนนี้มีแต่ forecast เพิ่มแถว "ฝนสะสมวันนี้จริงจากสถานี" เพื่อให้เห็นว่าพยากรณ์ต่ำ/สูงกว่าจริง
- **แถบเวลาน้ำขึ้น-ลง** (จาก sealevel API สสน.) สำหรับอำเภอติดทะเล — น้ำหลากเจอน้ำทะเลหนุนคือ scenario เลวร้ายสุดของ ต.ปะลุรู/ตากใบ

## 1.2 map.html — แผนที่สถานการณ์

**ที่มีอยู่:** 8 layers (ระดับน้ำ, ฝน, เขื่อน, ทะเล, CCTV, DDPM, telerid, พื้นที่เสี่ยง), flash flood report สสน., sparkline ใน popup, สลับภาษา, sidebar สรุป — เป็นหน้าที่แข็งแรงที่สุดของระบบ

**ควรเพิ่ม:**

- **📍 "ใกล้ฉัน" (geolocation)** — ปุ่มเดียว: ขอ GPS → ซูมไปตำแหน่ง → highlight สถานี+CCTV+จุดรายงานน้ำท่วมรัศมี 10 กม. ชาวบ้านใช้แผนที่แบบนี้ ไม่ได้ไล่ดูทีละ layer
- **⏱️ Time slider เรดาร์** — เรดาร์ย้อนหลัง 6 เฟรมเล่นเป็นอนิเมชัน ให้เห็น "ทิศทางฝนกำลังเคลื่อนมาทางไหน" (ภาพนิ่งบอกแค่ตอนนี้ อนิเมชันบอกอนาคต)
- **📤 ปุ่ม "แชร์สถานการณ์เข้ากลุ่มไลน์"** — สร้างภาพ snapshot สรุป (canvas: แผนที่ + ตัวเลขสถานีวิกฤต + timestamp) ให้ ปภ.อำเภอกดแชร์เข้ากลุ่มไลน์ราชการได้ใน 2 คลิก — ในไทย รายงานสถานการณ์เดินทางผ่าน LINE ไม่ใช่ผ่านเว็บ (มี copyReport อยู่แล้ว ต่อยอดเป็นภาพ)
- **ตัวกรอง "เฉพาะสถานีผิดปกติ"** — ช่วงวิกฤตมีสถานีเป็นร้อย toggle เดียวซ่อนสถานีเขียวทั้งหมด
- **Layer แนวคันกั้นน้ำ/ประตูระบายน้ำ + สถานะเปิด-ปิด** (ข้อมูลชลประทานนราธิวาสมีโครงการปริมาณมาก เช่น ปตร.บางนรา) — เพิ่มเป็น GeoJSON static ก่อน แล้วให้สมาชิกอัปเดตสถานะบานประตูผ่าน Supabase

## 1.3 forecast.html — คาดการณ์น้ำท่วม

**ที่มีอยู่:** คาดการณ์หลาย horizon (+6/12/24/72 ชม.) จาก trend extrapolation (calcRate), ดินอิ่มน้ำ, flash flood สสน., เขื่อนบางลาง, WRF-ROMS, แนวปฏิบัติ — แนวคิดถูกต้องและซื่อสัตย์กับข้อจำกัดของวิธีการ (แสดง confidence)

**ควรเพิ่ม:**

- **ต่อยอด Time-to-Bankfull ที่มีแล้ว** — ระบบคำนวณ "ถึงตลิ่ง ~X ชม." อยู่แล้ว (tta) แต่ควร (ก) แปลงเป็นเวลานาฬิกา "ราว 03:00 น. คืนนี้" — สมองมนุษย์ช่วงวิกฤตประมวล "ตี 3" ง่ายกว่า "+7 ชม." (ข) เพิ่มปุ่ม "คัดลอกข้อความแจ้งเตือน" ต่อสถานี — สร้างประโยครูปแบบโทรเลขราชการพร้อมส่งกลุ่มไลน์ทันที
- **เชื่อมฝนเข้าโมเดล** — trend อย่างเดียวพลาดกรณี "น้ำยังนิ่งแต่ฝนกำลังถล่มต้นน้ำ" เพิ่ม rule ง่าย: ถ้า WRF/nowcast คาดฝนต้นน้ำ >90 มม./วัน ให้ปรับ prediction band ขึ้นและติดป้าย "⚠️ ยังไม่รวมฝนที่กำลังจะตก คาดการณ์นี้เป็นขั้นต่ำ"
- **Scenario ปล่อยน้ำเขื่อนบางลาง** — slider "ถ้า กฟผ. เพิ่มการระบายเป็น X ล้าน ลบ.ม./วัน" → เวลาน้ำเดินทางถึงเมืองยะลา/ปัตตานีโดยประมาณ (ใช้ค่า travel time คงที่จากสถิติก็มีประโยชน์แล้ว)
- **ปุ่ม "เปิดในแบบจำลอง"** ต่อสถานี — deep link ไป sim.html พร้อมพิกัด + base level = ระดับคาดการณ์ (dtm รองรับ query param อยู่แล้ว) เชื่อมสองหน้าที่ตอนนี้แยกกันอยู่

## 1.4 history.html — น้ำท่วมย้อนหลัง (GISTDA)

**ที่มีอยู่:** แปลงน้ำท่วมจากดาวเทียมรายช่วงเวลา + สถิติรายจังหวัด

**ควรเพิ่ม:** โหมดเปรียบเทียบ 2 ช่วงเวลา (side-by-side หรือ swipe), overlay "น้ำท่วมครั้งใหญ่ล่าสุด" ลงบน map.html เป็น layer อ้างอิง, ตารางสถิติรายอำเภอ (ตอนนี้มีแค่จังหวัด) พร้อมปุ่ม export CSV ไว้แนบรายงานราชการ

## 1.5 repeat.html — พื้นที่เสี่ยงซ้ำซาก

**ที่มีอยู่:** flood-freq GISTDA + สรุปหมู่บ้านเสี่ยง + คลิกดูจุด

**ควรเพิ่ม:** **Gap Analysis อัตโนมัติ** — cross ข้อมูล 3 ชั้นที่มีอยู่แล้วในระบบ: หมู่บ้านเสี่ยงสูง × ระยะทางถึงศูนย์พักพิงใกล้สุด × จุดจอดจักรกล → ตาราง "หมู่บ้านเสี่ยงสูงที่อยู่ไกลศูนย์พักพิง >5 กม. และไม่มีเรือท้องแบนในรัศมี 15 กม." นี่คือเอกสารวางแผนก่อนฤดูฝนที่ ปภ.จังหวัดทำมือทุกปี ระบบคุณ generate ได้เลย

## 1.6 sim.html — แบบจำลองน้ำท่วม DEM 30m

**ที่มีอยู่:** iframe dtm + preset 5 จังหวัด + tiles preload — วิธีใช้เขียนดี ซื่อสัตย์เรื่องข้อจำกัด 30 ม.

**ควรเพิ่ม:** (ก) ปุ่ม "ใช้ระดับน้ำจริงตอนนี้" — fetch สถานีใกล้จุดโฟกัส แล้ว set base อัตโนมัติ แทนให้ผู้ใช้จำตัวเลขข้ามหน้า (ข) preset เพิ่มเติมจุดเสี่ยงประวัติศาสตร์: ตากใบ, ระแงะ, ศรีสาคร, เมืองนราธิวาส (ค) ปุ่มแชร์ URL scenario ปัจจุบัน

## 1.7 shelter.html — ศูนย์พักพิง ⚠️ หน้าที่ต้องยกเครื่องเร่งด่วนที่สุด

**ปัญหา:** ข้อมูล hard-coded เฉพาะนราธิวาส (`PROV_DATA = { 'นราธิวาส': DATA }`) ผู้ใช้สงขลา/ปัตตานี/ยะลา/สตูล เจอหน้าว่าง และไม่มีสถานะ real-time

**ยกเครื่องเป็น (Just Do It outline):**

ตาราง Supabase `shelters` (schema-v6):
```sql
create table public.shelters (
  id uuid primary key default gen_random_uuid(),
  name text not null, province text not null, amphoe text, tambon text,
  lat double precision, lon double precision,
  capacity int,                    -- ความจุ (คน)
  occupancy int default 0,         -- ผู้พักพิงปัจจุบัน
  status text default 'standby'    -- standby/open/full/closed
    check (status in ('standby','open','full','closed')),
  facilities jsonb,                -- {"kitchen":true,"medical":true,"pet":false,...}
  contact_name text, contact_phone text,
  updated_at timestamptz default now(), updated_by uuid
);
-- RLS: select สาธารณะ / update เฉพาะ approved member (ใช้ is_approved() เดิม)
```

UI layout:
```
┌──────────────┬────────────────────────────────┐
│ [ค้นหา/กรอง]  │            แผนที่               │
│ [📍ใกล้ฉัน]   │  marker สี: 🟢เปิด 🟡ใกล้เต็ม     │
│ ┌──────────┐ │           🔴เต็ม ⚪เตรียมพร้อม    │
│ │ศูนย์ A 🟢 │ │                                │
│ │ 120/300  │ │                                │
│ │ [นำทาง]  │ │                                │
│ │ [โทร]    │ │                                │
│ └──────────┘ │                                │
└──────────────┴────────────────────────────────┘
```

จุดสำคัญ: (ก) ปุ่ม **นำทาง** เปิด Google Maps directions ทันที (ข) ปุ่ม **โทร** เป็น `tel:` link — คนหนีน้ำใช้มือถือ (ค) สมาชิก approved เห็นปุ่ม "อัปเดตยอด" — แก้ occupancy/status ได้จากมือถือหน้างาน (ง) แถบความจุ `120/300 (40%)` แสดงเป็น progress bar

User flow ประชาชน: เปิดหน้า → กด "ใกล้ฉัน" → เห็นศูนย์เปิดที่ใกล้สุดที่ยังไม่เต็ม → กดนำทาง (3 แตะจบ)
User flow เจ้าหน้าที่ศูนย์: login → ค้นศูนย์ตัวเอง → กด +10 / -5 / เปลี่ยนสถานะ → เสร็จ

ข้อมูลตั้งต้น: ขอ list ศูนย์พักพิงจาก ปภ.จังหวัดทั้ง 5 (มีเป็น Excel อยู่แล้วทุกจังหวัด) — import ครั้งเดียว

## 1.8 route.html — เส้นทางน้ำท่วม

**ที่มีอยู่:** รายงานจุดน้ำท่วม/ถนนชำรุด (จุด/เส้น/รูปถ่าย/ความลึก/รถผ่านได้), Google Directions, ตรวจเส้นทางตัดกับจุดรายงาน, workflow แก้ไข/เคลียร์ — เป็นฟีเจอร์ที่ออกแบบดีมาก schema รัดกุม

**ควรเพิ่ม:**

- **Auto-stale** — รายงานอายุ >12 ชม. ที่ไม่มีใครยืนยัน แสดงจางลง + ป้าย "ยังไม่ยืนยันซ้ำ" พร้อมปุ่ม "ยังท่วมอยู่ ✓" ให้คนผ่านทางกดยืนยัน (นับ confirm ใน metadata) — ข้อมูล crowdsource เน่าเร็ว ระบบต้องแสดงอายุข้อมูล
- **ทางเลี่ยงอัตโนมัติ** — ระบบเช็คเส้นทาง × ชนิดรถ (VEHICLE_TYPES) มีแล้วและทำได้ดี ขั้นถัดไป: เมื่อเส้นทางไม่ผ่าน ให้ลองขอ Google Directions พร้อม `avoid` waypoint รอบจุดท่วมอัตโนมัติ แล้วเสนอ "เส้นทางเลี่ยง (+15 นาที)" แทนจบแค่ ❌
- **แจ้ง 1784** — ปุ่มลัดโทรสายด่วน ปภ. บนการ์ดรายงานถนนชำรุด
- **เชื่อม CCTV** — ถ้าจุดรายงานอยู่ใกล้กล้อง (layer มีใน map.js แล้ว) แสดงลิงก์ "ดูภาพสดจุดใกล้เคียง" ให้ verify ก่อนออกเดินทาง

## 1.9 resources.html — จักรกลสาธารณภัย

**ที่มีอยู่:** CRUD จักรกล (member-gated), ค้นหา, แผนที่, ข้อมูล deploy/demob, ผู้ประสานงาน — ครบสำหรับ "ทะเบียน"

**ควรเพิ่ม:** ยกจาก "ทะเบียน" เป็น "ระบบขอสนับสนุน":

- ตาราง `resource_requests`: หน่วยขอ, ชนิด/จำนวนที่ขอ, พื้นที่, ความเร่งด่วน, สถานะ (ขอ→อนุมัติ→ส่งแล้ว→คืนแล้ว) — ช่วงวิกฤต ปภ.จังหวัดรับโทรศัพท์ขอเรือทั้งวันโดยไม่มี tracking
- **สรุปกำลังพร้อมใช้** ด้านบน: `เรือท้องแบน: พร้อม 12 / ใช้งาน 8 / ซ่อม 2` ต่อชนิดต่อจังหวัด
- สถานะเครื่องจักรเพิ่ม `maintenance` และปุ่มเปลี่ยนสถานะเร็วจากมือถือ

## 1.10 admin.html — ผู้ดูแลระบบ

**ที่มีอยู่:** อนุมัติ/ปฏิเสธสมาชิก

## 2.1 🔔 alert.html + Notification Engine — **สำคัญอันดับ 1**

**ปัญหา:** ระบบทั้งหมดเป็น pull ผู้ใช้ต้องเปิดเว็บเอง

**สถาปัตยกรรม (ใช้ของที่มีอยู่ให้มากสุด):**
```
Supabase Edge Function (cron ทุก 15 นาที)
  → ดึง waterlevel API + nowcast API
  → เทียบกับ alert_rules
  → เข้าเงื่อนไข? insert alert_log + ยิง LINE Messaging API (OA) / Telegram Bot
```
หมายเหตุ: LINE Notify ปิดบริการแล้ว (มี.ค. 2025) — ใช้ LINE OA + Messaging API (broadcast ฟรี 200–300 ข้อความ/เดือน ต่อ OA — ตรวจโควตาปัจจุบันก่อนเลือก) หรือ Telegram Bot (ฟรีไม่จำกัด เหมาะกลุ่มข้าราชการ) แนะนำทำ Telegram ก่อนเพราะฟรีและง่าย แล้วค่อยเพิ่ม LINE OA สำหรับประชาชน

**ตาราง:**
```sql
create table alert_rules (
  id uuid primary key default gen_random_uuid(),
  station_code text,              -- หรือ null = ทั้งจังหวัด
  province text,
  metric text check (metric in ('wl_pct_bank','rain_24h','nowcast_intensity')),
  threshold numeric,              -- เช่น 80 (% ตลิ่ง), 90 (มม.)
  channel text,                   -- 'telegram:<chat_id>' / 'line:<group>'
  cooldown_min int default 180,   -- กันสแปม
  enabled boolean default true
);
create table alert_log (
  id uuid primary key default gen_random_uuid(),
  rule_id uuid references alert_rules(id),
  fired_at timestamptz default now(),
  payload jsonb                   -- ค่าที่ trigger + ข้อความที่ส่ง
);
```

**ตัวอย่างข้อความ:** `🟠 [เฝ้าระวัง] สถานี X037 บ.ท่าเรือ อ.สายบุรี ระดับน้ำ 85% ของตลิ่ง (+0.3 ม./3 ชม. กำลังขึ้น) คาดถึงตลิ่ง ~03:00 น. · ดูสด: waterchaidantai.com/map.html?st=X037`

**UI หน้า alert.html:** สาธารณะ = ปุ่ม "รับแจ้งเตือนอำเภอของฉัน" (QR เข้ากลุ่ม Telegram/LINE OA รายอำเภอ) · สมาชิก = ตารางจัดการ rules (สถานี / เกณฑ์ / ช่องทาง / เปิด-ปิด) + ประวัติการเตือน 50 รายการล่าสุด

**Effort:** Edge Function ~150 บรรทัด + หน้า UI 1 หน้า — ผลตอบแทนสูงสุดในทั้งเอกสารนี้

## 2.2 🎛️ eoc.html — War Room สำหรับข้าราชการ

**ปัญหา:** ผอ.ปภ. ถามว่า "สรุปตอนนี้เป็นไง" — คำตอบกระจายอยู่ 5 หน้า

**Layout (จอ TV war room / แท็บเล็ต ผอ.):**
```
┌───────────────────────────────────────────────────────┐
│ 🔴 สถานะรวม: เฝ้าระวังพิเศษ │ อัปเดต 14:30 │ [SITREP↓] │
├─────────────┬─────────────┬─────────────┬─────────────┤
│ สถานีวิกฤต   │ ฝน 24 ชม.    │ ศูนย์เปิด     │ รายงาน ปชช.  │
│  3 / 42     │ สูงสุด 120มม. │ 5 (312 คน)  │ 8 จุด active │
├─────────────┴─────────────┼─────────────┴─────────────┤
│ แผนที่รวม (สถานีแดง+จุดรายงาน │ ⏱️ Timeline เหตุการณ์      │
│  +ศูนย์เปิด — auto-refresh)   │ 14:10 X037 เกิน 80%      │
│                            │ 13:45 เปิดศูนย์ รร.บ้านทอน  │
│                            │ 13:20 รายงานถนน 4056 ท่วม  │
├────────────────────────────┴───────────────────────────┤
│ 📋 บันทึกเหตุการณ์/ข้อสั่งการ (incident log — สมาชิกพิมพ์เพิ่มได้) │
└────────────────────────────────────────────────────────┘
```

**ฟีเจอร์ฆาตกร: ปุ่ม SITREP** — generate รายงานสถานการณ์ประจำวันรูปแบบราชการ (docx/PDF) จากข้อมูลในระบบอัตโนมัติ: สถานการณ์น้ำ, พื้นที่ประสบภัย, ศูนย์พักพิง+ยอดผู้พักพิง, ทรัพยากรที่ deploy, เหตุการณ์สำคัญ — งานที่เจ้าหน้าที่ ปภ.นั่งพิมพ์มือทุกเย็น เหลือกดปุ่มเดียวแล้วตรวจแก้

**ข้อมูล:** ทุกอย่างมีในระบบแล้ว (waterlevel, flood_reports, machinery + shelters/incidents ใหม่) — หน้านี้คือการ "ประกอบ" ไม่ใช่สร้างใหม่ · ตารางใหม่ตารางเดียว: `incidents (id, ts, type, area, detail, created_by)`

## 2.3 📱 check.html — "เช็คพื้นที่ฉัน" สำหรับประชาชน

**ปัญหา:** Dashboard 10 หน้าเหมาะเจ้าหน้าที่ แต่ชาวบ้านที่กังวลต้องการคำตอบเดียว: "บ้านฉันจะท่วมไหม ต้องทำอะไร"

**UX: หน้าเดียว input เดียว** — กด "📍 ใช้ตำแหน่งฉัน" (หรือพิมพ์ตำบล) → ได้การ์ดตอบ 5 เรื่อง:

1. ระดับความเสี่ยงตอนนี้ (🟢🟡🟠🔴 + คำอธิบายภาษาบ้าน ๆ)
2. ฝนใน 3 ชม. (nowcast) + 24 ชม. ข้างหน้า
3. สถานีน้ำใกล้สุด + ระดับเทียบตลิ่ง + แนวโน้ม
4. พื้นที่นี้เป็นจุดท่วมซ้ำซากหรือไม่ (flood-freq ที่จุดนั้น)
5. ศูนย์พักพิงใกล้สุดที่เปิด + ปุ่มนำทาง + ปุ่มรับแจ้งเตือน

ทุก data source มีอยู่แล้วในระบบ — หน้านี้คือ "composition layer" ~1 วันงาน แต่เป็นหน้าที่ควรอยู่บนโปสเตอร์ QR code ติดศาลาหมู่บ้าน และเป็น URL ที่กำนัน/ผู้ใหญ่บ้านแชร์เข้ากลุ่มไลน์หมู่บ้าน

## 2.4 🧓 กลุ่มเปราะบาง (people.html) — เฟสถัดไป

ทะเบียนผู้ป่วยติดเตียง/ผู้สูงอายุ/ผู้พิการในพื้นที่เสี่ยง เพื่อวางแผนอพยพ — มีมูลค่าสูงมากต่อ ปภ. แต่เป็นข้อมูลอ่อนไหว (ข้อมูลสุขภาพ + PDPA) ต้องทำ RLS เข้มงวด (เห็นเฉพาะ admin จังหวัดตัวเอง), เก็บเท่าที่จำเป็น, และควรประสานรูปแบบข้อมูลกับ อบต./รพ.สต. ก่อน — แนะนำทำหลังระบบหลักนิ่งแล้ว

## 2.5 📴 PWA + Offline Mode

น้ำท่วมใหญ่ = ไฟดับ + เสาสัญญาณล่ม เพิ่ม manifest.json + service worker: cache หน้า + ข้อมูลล่าสุดที่ดึงสำเร็จ พร้อมประทับ "⚠️ ข้อมูล ณ 14:30 (ออฟไลน์อยู่)" — อย่างน้อยผู้ใช้ยังเห็นศูนย์พักพิง เบอร์โทร และข้อมูลชุดสุดท้ายได้ และ "Add to Home Screen" ทำให้เว็บกลายเป็นแอปในมือถือชาวบ้าน
